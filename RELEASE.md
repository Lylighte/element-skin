# Element Skin v3.0.0

> **重大版本升级 / BREAKING CHANGES**
>
> v3.0.0 是一次后端、权限模型、API 和部署方式的整体升级。升级前请完整备份 PostgreSQL 数据库、Redis 中仍需保留的运行数据、配置文件、材质文件以及站点签名密钥。
>
> 当前版本只保留从 **v2.4.1** PostgreSQL 数据库升级到 v3.0.0 的迁移路径。请先将现有站点升级到 v2.4.1，再执行 v3.0.0 升级；开发期间的中间版本不属于受支持的迁移来源。

## 新功能

### 后端与部署

- 后端从 Python 重构为 Go 1.26，采用路由层、业务 service 层、数据库 store 层和 Redis store 层的分层结构。
- 使用 PostgreSQL 18 作为持久化数据库，使用 Redis 保存短期 token、鉴权缓存、验证码、限流数据、公开数据缓存和调度任务状态。
- Docker 部署统一使用根目录的 `docker-compose.yml` 和 `.env`，不再需要维护前后端分离的部署方式或额外的后端配置副本。
- 单镜像同时提供 Go 后端和构建后的前端静态资源，支持通过环境变量配置站点 URL、API URL、前端路径、API 路径、数据库、Redis 和 CORS。
- 启动时自动生成并持久化站点签名密钥。已有部署必须保留 `data` 目录中的密钥，否则 Yggdrasil 客户端会看到服务端签名身份变化。

### 细粒度权限模型

- 站点所有受保护能力统一使用细粒度权限，不再使用 `user`、`admin` 或 `super_admin` 作为最终授权判断。
- 权限主体可以同时拥有多个角色，并可通过单项权限覆盖进行精细调整。
- 权限支持 `self`、`own`、`bound_profile`、`any` 等资源范围，分别表达当前主体、主体拥有的资源、主体绑定的角色资源和全局资源。
- 受保护主体从角色概念中独立出来。站点最多有一个受保护主体，具有相应权限的主体可以将受保护状态转让给其他用户；受保护状态不是绕过权限检查的超级管理员开关。
- 管理后台和前端导航根据细粒度权限显示功能入口；后端 service 仍然负责最终权限校验。
- 用户角色、单项权限、权限覆盖和受保护主体变更会触发对应的系统通知及 OAuth 依赖关系处理。

### OAuth 与第三方应用

- 提供统一的 `/v1` 站点 API，Cookie 会话和 OAuth Bearer token 使用同一套业务能力与细粒度权限模型。
- 支持 OAuth Authorization Code + PKCE、Device Authorization Grant 和 Client Credentials Grant。
- 支持公开应用和机密应用；机密应用使用 client secret，公开应用不依赖 client secret。
- 支持第三方应用创建、审核、驳回、停用、重新提交审核和密钥轮换。
- 应用拥有独立的权限主体。应用权限由管理员审核后的权限集合决定，Client Credentials 只能使用应用本身具备的权限，不能借用某个用户的 Cookie 或用户授权。
- 用户可以逐项授权第三方应用访问自己有权授予的能力；授权记录、refresh token、access token、授权码和设备码拥有独立生命周期。
- 撤销用户授权、应用停用、应用驳回、应用密钥轮换、授权者权限变化和应用创建者权限变化会按对应触发条件失效关联授权或应用。
- 提供 Python SDK，包含 API 客户端、OAuth 流程、PKCE、token 存储、权限模型和中文开发文档。

### 通知与公告

- 引入统一通知中心，公告只是通知系统中的一种类型，后续可以承载系统通知和 OAuth 通知。
- 支持短公告和长公告：长公告使用摘要和 Markdown 正文，短公告只展示短内容，不要求正文 Markdown。
- 支持面向所有用户、管理员和指定用户的通知投递；通知中心提供未读角标、列表和详情展示。
- OAuth 应用审核结果、应用停用或驳回、权限变化、授权失效、账号封禁等系统事件可以向对应用户发送通知。
- 支持置顶、忽略、已读、有效开始时间和有效结束时间；系统通知默认有效 30 天，过期通知由统一调度任务清理。
- 管理员更新公告采用删除旧公告并发布新公告的语义，避免历史内容与已读状态产生歧义。

### Yggdrasil 与第三方服务

- 保留现有 Yggdrasil 协议端点，Yggdrasil 路径不迁移到 `/v1`，继续服务启动器、游戏客户端和服务器。
- 根发现入口、元数据和 `/api/publickeys/` 可以聚合本站及 fallback 服务的公钥，并对 fallback 公钥进行 Redis 缓存、去重和失效处理。
- fallback 的 Services 健康检查继续保持原有检查逻辑；公钥发现与健康检查分离。
- 从第三方皮肤站导入角色时同步下载并保存角色材质，避免只导入角色元数据而缺少材质文件。
- Yggdrasil 账号、会话、加入服务器和材质签名路径使用 Redis 保存短期状态，并继续支持 Authlib-Injector。

### 前端体验

- 重构仪表盘、通知中心、OAuth 应用、管理员用户与权限管理等页面，统一使用现有 UI 组件和布局系统。
- 仪表盘展示资源统计、快速接入启动器、服务状态和右侧公告列；窄屏自动切换为单列布局。
- 第三方应用页面同时提供应用开发者管理和用户授权管理，应用详情按需加载。
- 新增 OAuth 授权确认页，Authorization Code 流和 Device Code 流共用同一套授权确认组件。
- 管理员可以在用户详情中查看和编辑角色、继承权限、单项覆盖及受保护主体状态。
- 支持个人资料中的邮箱变更弹窗、验证码校验以及登录注册页面对可选邀请码字段的正确展示。
- 顶栏导航会根据可用空间逐项收起，通知中心在存在未读通知时显示小红点。
- 皮肤库的使用次数改为紧凑的图标与数字展示，角色和材质静态渲染结果使用浏览器缓存抽象并受 LRU 与总大小限制。

## API 与数据库变更

- 站点 API 统一迁移到 `/v1` 命名空间；Yggdrasil 协议端点保持原路径不变。
- 新增 OAuth、权限、通知、应用管理、公开配置、邮箱变更、fallback 和 Minecraft 相关的 `/v1` 接口，完整接口定义见 `doc/Element-Skin-v1-API设计规范.md`。
- API 错误响应统一使用结构化 `detail` 字段；OAuth 标准端点使用 OAuth 规定的错误格式；Yggdrasil 错误保持协议格式。
- v2.4.1 的用户、角色、材质、fallback、设置和验证数据会迁移到当前结构。
- 迁移会删除不再使用的旧 `tokens` 和 `sessions` 数据表；短期认证和会话状态由 Redis 管理。
- 迁移会将 v2.4.1 的管理员信息转换为细粒度权限模型，并建立受保护主体状态。升级后请检查管理员角色、权限覆盖和受保护主体。
- 权限位索引是运行时编码信息，不作为数据库字段或持久化业务数据保存。

## 升级步骤

1. 停止当前站点并备份 PostgreSQL 数据库。
2. 备份根目录 `.env`、`data` 目录、材质文件目录和站点签名密钥。
3. 确认当前数据库来源是 v2.4.1；不要从开发期间的中间数据库直接升级。
4. 准备 Redis，并确认生产环境显式提供数据库、Redis、站点 URL、API URL、JWT secret 和 CORS 等必需配置。
5. 更新镜像和 Compose 文件：

   ```bash
   docker compose pull
   docker compose up -d
   ```

6. 查看后端启动日志，确认数据库迁移、Redis 连接和签名密钥加载均成功。
7. 登录站点并检查管理员权限、受保护主体、用户角色、邀请码、材质、fallback、OAuth 应用和通知中心。
8. 重新检查 Nginx 的前端静态目录、API 反向代理和 `X-Authlib-Injector-API-Location` 配置。
9. 使用测试账号验证 Cookie 登录、OAuth token、Yggdrasil 登录和材质下载链路。

## 回滚说明

- v3.0.0 的数据库迁移不是可逆迁移。需要回滚时，停止 v3.0.0 服务并从升级前的 PostgreSQL 备份、配置和数据目录恢复，再部署 v2.4.1。
- 不要直接让旧版本连接已经完成 v3.0.0 迁移的数据库。
- 不要删除或替换已有站点签名密钥；密钥变化会使客户端验证到不同的服务端身份。

## 验证与质量

- Go 后端提供数据库层、业务层、HTTP 集成层和真实 Redis 集成测试。
- 关键权限、OAuth、token 轮换、通知、缓存、迁移、导入和 Yggdrasil 路径包含精确断言及错误路径测试。
- SDK 测试覆盖 API 请求、OAuth 流程、PKCE、token 存储、权限校验和异常分支。
- 当前仓库包含 Cookie、OAuth 以及混合并发场景的压测报告，详见 `reports/`。
